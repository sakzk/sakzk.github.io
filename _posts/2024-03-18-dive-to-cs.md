# 記録
### 2024-03-19
- オペレーティングシステム: 3章
	- mutexがふつうにむずい。脳がちぎれるほどはむずかしくないが、じくじくする程度にはむずい。
	- 一番単純なシチュエーションを考える。すなわち、ユニプロセッサ・2プロセス。物理CPUからは、複数のプログラムをマッシュアップした一連の機械語を実行しているように見えていている。レジスタ、メモリ空間、I/Oを分離しないとごちゃごちゃになるよね、というのが前提。
		- メモリ、I/Oの仮想化はいったん忘れる
		- 話が複雑になるので、マルチコアとか、キャッシュとかはいったん忘れる
		- 実行可能状態プロセスの待ち行列、スケジューリング方式とかも話が難しくなるのでいったん忘れる
		- 実行中プロセスへの割り込みが切り替えのきっかけだが、割り込みは排他制御そのものではない。
			- 切り替えてもプロセスが競合、干渉しないためにどうするか、というのが排他制御。
	- 原始的に実行できる機械語命令があること仮定する。
		- これをいい感じに組み合わせて、競合を解決する
			- これがない状況はいったん忘れる (そういうアーキテクチャもあるかもしれないが忘れる)
		- 機械語レベルのことをやっているのか、言語機能レベルのことをやっているのか区別する
	- マルチプロセスで生じる問題と解決のための仕組みをざっと読む。問題の単純さはどの程度か、どの仕組みでどこまで解決できるか、というスコープを持って読む。メモリ上のデータにアクセスするのは、メモリ空間別なので競合しないのではとか疑問が浮かぶので、I/Oの競合 (ex. ファイルの読み書き) を想定しながら読む。
		- よくかんがえたらI/Oの仮想化もよくわからんので、2つのプロセスに共有メモリがあるとして進めるのがよさそう。
	- 他：なんとなく全体を流し読みしてみたが、mutex、セグメンテーションのアドレス変換が一番パズル的な難しさがある。それ以外はわりと行けるっぽい。

- Code Complete: 6, 7部
	- 大きなものは設計が大変になる (要求を明らかにするのはむずい、というのは正直イメージがつかない)
		- 組み合わせ爆発
	- 統合戦略は、いかにしてつくるか、に直結する。
		- 「どう作るか」にうまくいく万能のやり方はない。複雑性の累積が低くなるやり方を、制約に応じて選ぶ。
		- 複雑性、難しさに対するカンを大事にする。問題特有の本質的にな難しさ、ややこしさ以外は0であるべき。
			- 本質的な難しさとは、コンピュータができることと自動化したいことのギャップ。手順、依存、例外／エンドユーザー、社会の複雑さ。
			- 自分がどのレベルの抽象度で作業しているのか把握してできるだけ上で作業する。
				- 下の層で抽象を作って上の層に提供する。上の層の変更がソースコードの変更に影響しづらいように部品を組み立てていく。
				- 問題領域の表現／下位レベルの問題(ビジネス層・サービス層)／下位レベルの実装(ADS)／言語(ライブラリ・文法・イディオム)／機械語・OS
	- ソフトウェアエンジニアリングで発生するソースコード上の意味的な操作を節約するためにツール類がある。
		- 何の手間をなくしたいのか？ プロジェクトの規模に応じたツールの選択、発展的乗り換え。
		- 機械的に反復していることを実現するツール、やり方があるはずだ、という視点で探す。
		- 本質的に反復ではないことに頭を使う。
			- 問題特有の処理を理解してコードで表現すること
		- ポチッと押すだけでできると、チクチク手作業でやるの間のどこで問題を解決するか。自動化、省エネ化の労力の元が取れる範囲はどこまでか決めないと yak shavingになりがち。
	- ソースコードを、単に書くもの以外のものとして扱う。
		- 読んでわかるもの、変えやすいもの
	- どちらでもよいものを固定して余計な意味を持たせないのがコーディング規約。
	- 心構え
		- 「間違いは複雑な頭脳労働の浮き沈みの一端として受け入れられる。Sallyが手抜きをしない限り、だれも彼女の失敗を責めたりしない」
		- 解決されなかったらそれは失敗。解決に至る試行錯誤、反復は失敗ではない。
		- コンパイルできないのは問題だが、コンパイルしないと分からないのも問題。
